# CLAUDE.md — Python ETL Pipeline

## Project Overview

A Flask-based ETL platform that allows users to upload CSV files, explore and validate the data interactively before committing it to a PostgreSQL database, and schedule recurring upload jobs. The UI also provides a job monitoring tab.

---

## Tech Stack

| Layer | Tool | Rationale |
|---|---|---|
| Web Framework | Flask | Lightweight, sufficient for this scope |
| Database | PostgreSQL (Docker) | Reliable, strongly typed, easy to run locally |
| ORM / DB Layer | SQLAlchemy + psycopg2-binary | Standard Python ORM; raw queries allowed for ETL |
| Migrations | Flask-Migrate (Alembic) | Tracks schema changes; never alter tables manually |
| Data Exploration | Pandas | CSV parsing, profiling, type inference |
| Job Scheduling | APScheduler (BackgroundScheduler) | No broker required; runs in-process |
| Containerization | Docker + Docker Compose | Postgres + app services defined together |
| Config | python-dotenv (.env) | No hardcoded credentials anywhere |

## Data Flow

```
CSV Upload
    └─► csv_processor.py
            ├── Parse with pandas (dtype=str initially)
            ├── Profile: shape, nulls, unique values, inferred types
            └── Store parsed DataFrame in session or temp file

Explore Tab (user reviews profile)
    └─► User confirms or adjusts column types / null handling
            └─► POST to /explore/confirm

db_manager.py
    ├── Infer or receive column type mapping
    ├── CREATE TABLE IF NOT EXISTS with inferred schema
    └── Bulk insert using SQLAlchemy Core (not ORM) for performance

ETLJob (scheduled)
    └─► APScheduler triggers a registered function
            ├── Reads source CSV (local path or configured source)
            ├── Runs through csv_processor → db_manager pipeline
            └── Writes result to UploadLog; updates ETLJob.last_run + status
```

## Models

### ETLJob

```python
class ETLJob(db.Model):
    id           = db.Column(db.Integer, primary_key=True)
    process_name = db.Column(db.String(120), nullable=False)
    schedule     = db.Column(db.String(80), nullable=False)  # cron string or interval
    status       = db.Column(db.String(20), default="idle")  # idle | running | failed | success
    last_run     = db.Column(db.DateTime, nullable=True)
    source_path  = db.Column(db.String(255), nullable=True)  # file path for scheduled jobs
    target_table = db.Column(db.String(120), nullable=True)
    created_at   = db.Column(db.DateTime, default=datetime.utcnow)
```

### UploadLog

```python
class UploadLog(db.Model):
    id            = db.Column(db.Integer, primary_key=True)
    filename      = db.Column(db.String(255))
    target_table  = db.Column(db.String(120))
    rows_inserted = db.Column(db.Integer, default=0)
    rows_failed   = db.Column(db.Integer, default=0)
    error_message = db.Column(db.Text, nullable=True)
    triggered_by  = db.Column(db.String(40))  # "manual" | "scheduler"
    created_at    = db.Column(db.DateTime, default=datetime.utcnow)
```

## Scheduler Rules

- Use `APScheduler.BackgroundScheduler` initialized in `extensions.py`, started in `create_app()`.
- Jobs are loaded from the `ETLJob` table at startup and re-registered.
- Schedule format stored in `ETLJob.schedule` must follow cron syntax: `"0 6 * * *"` (daily at 6am).
- Interval-based jobs use keyword `"interval:minutes=30"` — parse the prefix to decide trigger type.
- On job execution: set `status = "running"` → run pipeline → set `status = "success"` or `"failed"`.
- **Never** modify `APScheduler` state directly from a route — always go through `services/scheduler.py`.

---

## CSV Processor Rules

- Always read CSV with `dtype=str` first to prevent Pandas type coercion surprises.
- Profile step must report: row count, column names, null counts per column, inferred target dtype.
- Type inference priority: `int64 → float64 → datetime → bool → str`.
- Dates: attempt ISO 8601 first (`%Y-%m-%d`), then common regional formats. Store inferred format; show it to user in Explore tab.
- Strip leading/trailing whitespace from all string columns before insert.
- If a column is >95% null, flag it as a warning in the Explore tab — do not automatically drop it.

---

## DB Manager Rules

- Use `SQLAlchemy Core` (`Table`, `Column`, `insert()`) for bulk inserts — not the ORM — for performance.
- `CREATE TABLE IF NOT EXISTS` — never `DROP TABLE`. For schema changes, use migrations.
- Chunk inserts: max 1000 rows per `execute()` call.
- Wrap each batch in a transaction; roll back the batch on error and log to `UploadLog.rows_failed`.
- Column name sanitization: lowercase, replace spaces with underscores, strip special characters.
- Do not create tables with column names that are PostgreSQL reserved keywords (check against a defined list).

## Flask UI — Tab Structure

```
[ Upload ] [ Explore ] [ Jobs ]
```

**Upload tab**: File picker (CSV only), target table name input, submit. Redirects to Explore after upload.

**Explore tab**: Shows a summary table (column | inferred type | null% | sample values). User can override types via dropdowns. "Confirm & Insert" button triggers DB write. Shows insert result (rows inserted / errors).

**Jobs tab**: Table with columns `ID | Process Name | Schedule | Status | Last Run | Actions`. Status uses color badges (idle=gray, running=blue, success=green, failed=red). Actions: manual trigger button per row.

## Error Handling

- All routes return JSON errors for AJAX calls (`{"error": "message", "status": 400}`).
- CSV parsing errors must be caught and shown in the UI — never crash silently.
- DB insert errors: log to `UploadLog`, surface in the Explore tab response.
- Scheduler errors: update `ETLJob.status = "failed"`, log traceback to `UploadLog.error_message`.

## Testing Guidelines

- Use `pytest` with `pytest-flask` and a separate test database (configured via `FLASK_ENV=testing`).
- `csv_processor.py`: unit-test type inference with edge case CSVs (mixed types, nulls, encodings).
- `db_manager.py`: integration tests against a real test Postgres container (use `pytest-docker` or a fixture that spins up the compose service).
- `scheduler.py`: test job registration and status transitions with mocked APScheduler.
- Never mock the database in integration tests — use a real connection to a test schema.

---

## What NOT to Do

- Do not use `pd.read_sql` for bulk inserts — use SQLAlchemy Core.
- Do not store credentials in code or commit `.env`.
- Do not use `DROP TABLE` in any service — use `IF NOT EXISTS` and migrations for schema evolution.
- Do not schedule jobs by storing raw Python `datetime` — always use cron strings so they survive restarts.
- Do not run `flask db upgrade` inside application code — it is a CLI-only operation.
- Do not use Flask's built-in session to store DataFrames — serialize to a temp file under `uploads/`.
